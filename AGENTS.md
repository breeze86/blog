# AGENTS.md

## 1. 沟通规则

- 默认使用简体中文。
- 先给结论，再给分析。
- 回答务实、直接，避免空泛描述。
- 不确定内容必须明确标注“待确认”，禁止猜测或编造。

## 2. 工作方式

- 先分析后实现：先给修改方案，待确认后再动代码。
- 若用户明确说“直接改 / 立即开发 / 现在就做”，可跳过确认直接执行。
- 优先最小必要改动，避免无关重构。
- 优先复用现有目录结构、命名方式、组件模式、样式体系。
- 非必要不引入新依赖或复杂抽象。

## 3. 项目定位与默认目标

- 当前项目是个人博客站点，不是通用产品后台。
- 第一优先级：内容发布体验、阅读体验、加载速度、SEO、可维护性。
- 第二优先级：个性化展示（主页、侧栏、字体、壁纸、动效等）。
- 对“炫技功能”保持克制，避免牺牲性能与可读性。

## 4. Firefly 主题微调原则

- 优先改 `src/config/*` 配置，不优先改主题核心组件。
- 常规定制优先顺序：
  1. `siteConfig.ts`（站点信息/页面开关/SEO/布局）
  2. `profileConfig.ts`（头像/简介/社交链接）
  3. `navBarConfig.ts`（导航结构）
  4. `sidebarConfig.ts`（侧栏与移动端底部组件）
  5. 其他模块化配置（评论、页脚、字体、壁纸、友链、相册等）
- 必须改主题实现时：
  - 先说明为什么不能通过配置完成；
  - 改动范围最小化；
  - 标记“主题差异点”，便于后续同步上游主题版本。

## 5. 代码与目录约束

- 使用 TypeScript，避免滥用 `any`。
- 沿用现有技术栈（Astro + Svelte + Tailwind + Stylus）。
- 页面与内容结构遵循现有目录：
  - 页面：`src/pages`
  - 组件：`src/components`
  - 配置：`src/config`
  - 内容：`src/content/posts`
  - 样式：`src/styles`
- 不擅自删除文件、修改公共接口、升级大版本依赖。
- 不引入 axios；HTTP 请求优先 `fetch` 或项目现有方案。
- 外部数据默认不可信，必须做校验和兜底。

## 6. 博客内容规范（Frontmatter）

- 新文章优先使用现有字段：`title`、`published`、`description`、`tags`、`category`、`draft`、`comment`、`pinned`。
- `draft` 在发布前保持 `true`，发布时改为 `false`。
- 图片路径优先使用项目既有规范，避免混乱引用。
- 分类与标签命名保持稳定，避免同义词碎片化。

## 7. 页面与功能开关策略

- 通过 `siteConfig.pages` 控制 `friends/sponsor/guestbook/bangumi/gallery` 页面启停。
- 关闭页面时同步检查导航入口（`navBarConfig.ts`）与 sitemap 行为。
- 非核心功能默认按“个人博客最小可用”原则启用，避免首屏负担过大。

## 8. 性能与SEO基线

- 所有改动默认考虑静态生成场景（SSG）与构建时行为。
- 优先保证：
  - 构建成功与页面可访问性；
  - 元信息完整（title/description/keywords/og）；
  - 图片与脚本体积可控；
  - 移动端可用性。
- 涉及图片、字体、动画、播放器、看板娘等功能时，评估性能影响后再启用。

## 9. 质量检查与验证

- 改动后优先执行：
  - `pnpm lint`
  - `pnpm type-check`
  - `pnpm check`
  - `pnpm build`
- 若某项未执行，必须明确说明原因和潜在风险。
- 禁止伪造测试结果、构建结果或运行结果。

## 10. 变更说明模板（每次改动后必须给）

1. 改了什么（文件 + 行为）
2. 为什么改（目标与约束）
3. 影响范围（页面/组件/SEO/性能/构建）
4. 验证结果（执行命令与结果）
5. 未覆盖风险与后续建议（如有）

## 11. 明确禁止事项

- 未确认前做大规模重构。
- 为了“好看”改动核心结构导致后续难以升级。
- 引入与个人博客目标无关的复杂系统设计。
- 在未评估影响前开启重资源功能（大量动效、重脚本组件等）。

## 12. 上游主题同步策略（Firefly）

- 本地定制与主题核心尽量解耦（配置覆盖优先）。
- 记录所有核心改动点，形成差异清单。
- 同步上游版本时按“先比对、再合并、后回归验证”执行。
